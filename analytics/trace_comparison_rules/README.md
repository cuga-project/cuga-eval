# Trace Comparison Analytics

Analyzes Langfuse execution traces from evaluation bundles to explain **why** an agent fails on tasks it sometimes passes. It compares successful and failed runs step-by-step, clusters the failure patterns, identifies root causes, and recommends prioritized remediation actions.

## How It Works

1. **Step-by-step comparison** — For each (successful, failed) trace pair, compares agent outputs field-by-field at every step using comparison rules generated by `comparison_designer.py`. Produces a markdown report (`step_comparison.md`).

2. **Root cause analysis** — Feeds the comparison report to an LLM that clusters failures, explains root causes, and recommends fixes. Produces a markdown report (`summary.md`).

Each pipeline run writes its outputs to a timestamped folder:
```
benchmarks/{benchmark}/analytics_output/trace_comparison_rules/
  {timestamp}_{benchmark}_{agent_config}/
    config.json           ← experiment configuration and pair index
    step_comparison.md    ← step-by-step comparison report
    summary.md            ← root cause analysis and recommendations
```

## Prerequisites

- Evaluation bundles with Langfuse traces in `benchmarks/{benchmark}/evaluation_bundles/`
  (produced by `compare.sh` runs; requires Langfuse tracing to be enabled)
- API access to an LLM for the analysis steps. The pipeline uses [LiteLLM](https://docs.litellm.ai/) as a routing layer, so any provider LiteLLM supports works: OpenAI, Azure OpenAI, Anthropic, AWS Bedrock, and others. The provider is inferred from the model name prefix (e.g. `azure/gpt-4.1`, `aws/claude-opus-4-6`, `aws/claude-sonnet-4-5`). Configure credentials in `.env`:

  ```env
  # Option A: direct provider credentials (LiteLLM picks up standard env vars)
  OPENAI_API_KEY=...          # for openai/* or azure/* models
  AZURE_API_KEY=...           # for azure/* models
  ANTHROPIC_API_KEY=...       # for anthropic/* models

  # Option B: LiteLLM proxy (e.g. an IBM internal gateway)
  LITELLM_API_KEY=...         # or falls back to OPENAI_API_KEY
  LITELLM_BASE_URL=...        # or falls back to OPENAI_BASE_URL
  ```

  Set `MODEL` in your config file (or pass `--model`) to match your credentials.

## Running via Shell Script

Use `scripts/analyze.sh` (top-level dispatcher) or the benchmark-specific shortcut:

```bash
# Top-level dispatcher — works for any benchmark
./scripts/analyze.sh --benchmark appworld --analytics trace_compare
./scripts/analyze.sh --benchmark m3 --analytics trace_compare

# Benchmark shortcuts (forward all args to scripts/analyze.sh)
./benchmarks/appworld/analyze.sh --analytics trace_compare
./benchmarks/m3/analyze.sh --analytics trace_compare
```

Common options:

```bash
# Use a custom config file
./scripts/analyze.sh --benchmark appworld --analytics trace_compare --config my_experiment.conf

# Override specific parameters
./scripts/analyze.sh --benchmark appworld --analytics trace_compare \
  --bundles 20260427_120630_compare_gpt-oss_nosdk_1task_10runs \
  --pairing-mode all_trajectories

# Filter to specific tasks
./scripts/analyze.sh --benchmark appworld --analytics trace_compare \
  --task-ids e775c78_1 fd1f8fa_2

# Analyze bundles from the last two weeks
./scripts/analyze.sh --benchmark appworld --analytics trace_compare \
  --since 2026-04-15
```

## Configuration

Default parameters live in `config/default.conf`. Create additional `.conf` files in the same folder for different experiment setups and select with `--config`.

```bash
# config/default.conf
BENCHMARK="appworld"           # benchmark
AGENT_CONFIG="cuga_lite_top"   # subfolder under agent_prompts/
RULES="2026-05-03_16-24-50_cuga_lite_top_comparisons.txt"  # rules file in comparison_rules/, or 'latest'
BUNDLES="all"                   # 'all' or space-separated bundle folder names
PAIRING_MODE="one_pair_per_task"
N=5                             # used only with n_pairs pairing mode
MODEL="aws/claude-opus-4-6"
PROMPT_FILE="root_cause_analysis.jinja"
MAX_PAIRS=50                    # cap for all_permutations / all_trajectories; 0 to disable
SINCE=""                        # e.g. 2026-04-01 — leave empty to disable
TASK_IDS=""                     # leave empty to include all tasks
```

### Pairing modes

| Mode | Description |
|---|---|
| `one_pair_per_task` | One random (success, fail) pair per task |
| `n_pairs` | N pairs total across tasks (set `N` in config or `--n`) |
| `all_trajectories` | Every trace participates in at least one pair per task; lists are shuffled and zipped with recycling of the shorter list |
| `all_permutations` | Every (success, fail) combination per task |

`all_permutations` and `all_trajectories` are capped at **50 pairs by default** to control cost and context length. Override with `--max-pairs N` (or set `MAX_PAIRS=N` in a config file). Set to `0` to disable the cap entirely.


## Agent Configurations

Agent prompt configurations live in `agent_prompts/`. Each subfolder represents one agent configuration and contains:
- `cuga_agent_manifest.json` — maps CUGA sub-agent names (Task Analyzer, Task Decomposition, Plan Controller, Code Agent, Reflection, Shortlister, Final Answer) to their `.jinja2` system-prompt files. These names are tightly coupled to the CUGA agent stack; adding support for a different agent architecture would require a new subfolder with its own manifest.
- `*.jinja2` — system prompt templates for each agent

## Comparison Rules

Pre-generated comparison rules files live in `comparison_rules/`. Rules are generated by `comparison_designer.py` and define the output schema and comparison mode (exact match, fuzzy, semantic similarity, etc.) for each agent's output fields.

To generate new rules for an agent configuration:
```bash
cd analytics/trace_comparison_rules
uv run --no-sync python src/comparison_designer.py \
  --agent-config cuga_lite_top \
  --model aws/claude-opus-4-6
```
